## 的

このリポジトリでは、**変更の実装はCodex、運用はスクリプトとCI**で回す。

人間の作業を減らしつつ、品質ゲート（lint / build / E2E / visual）を必ず通す。

## 絶対ルール

1. **main へ直接コミット禁止**。必ず新規ブランチで作業する。
2. 変更は **最小差分**。目的に関係ない整形・リネーム・大規模改修は禁止。
3. **秘密情報をコミットしない**（`.env*`、APIキー、トークン、個人情報、ログの生データ）。
4. 変更後は必ず **ローカルで検証してから**終了する（下記「必須チェック」）。
5. CIが落ちたら、**最短で通す修正**を行う（根本改修でなく原因潰し）。

## 作業フロー（基本）

1. 作業は `dev/pr.sh "作業内容"` で開始する（ブランチ作成〜PRまでのレール）。
2. Codexは、スクリプト実行中に求められる「実装」パートを担当する。
3. 完了時にスクリプトが走らせるチェックで落ちたら、Codexが修正する。
4. PR作成後は、GitHub Actions の `CI` が緑になることを確認する。
5. CI失敗時は `codex-autofix` が修正PRを作る。まずはそのPRを優先して確認する。

## 必須チェック（ローカル）

作業完了前に必ず実行して全て成功させる：

- `pnpm lint`
- `pnpm build`
- `pnpm exec playwright test`（E2E）

### Visual regression（導入済みの場合）

- `tests/visual.spec.ts` が存在する場合は必ず実行する
    - 通常：`pnpm exec playwright test tests/visual.spec.ts`
    - 意図したUI変更でベースライン更新が必要：`pnpm exec playwright test tests/visual.spec.ts --update-snapshots`
- **スナップショット更新は「意図したUI変更」のときだけ**行う。むやみに更新しない。

## ブランチ命名（推奨）

- `codex/YYYYMMDD-HHMM-<短い説明>`
    
    （`dev/pr.sh` が自動で作る形式に合わせる）
    

## PRの作法

- PR本文には最低限を記載：
    - 目的（何を直した/足した）
    - 変更点（主要ファイル）
    - 動作確認（通したコマンド）
    - UI変更がある場合：visual差分の意図説明
- Auto-mergeが有効なら `gh pr merge --auto --squash` を使う（スクリプト側で実行）。

## 禁止事項（よくある事故）

- `.env` を追加/変更してコミット
- lockfile を無関係に更新
- E2Eを不安定化させる `waitForTimeout` 乱用
- Visual差分が出たのに理由説明なしでスナップショットを更新

## セッションログ（任意だが推奨）

各作業の最後に `docs/ai/codex/` に **要約ログ**を追記する（生ログは残さない）：

- 何をしたか
- 何を決めたか
- TODO
- ハマりどころと解決
